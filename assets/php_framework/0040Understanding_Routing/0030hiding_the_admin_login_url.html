Hiding The Admin Login URL — Deprecated

Note: This technique belongs to older versions of Trongate. The login module (shipped with Trongate v2) replaces this approach entirely. Security through obscurity is no longer recommended — the login module is secure by design.

The Old Approach

In previous versions of Trongate, the trongate_administrators module featured a configurable $secret_login_segment property. Administrators could set a private word that acted as the login URL, making it harder for attackers to find the login page. Custom routing mapped this secret word to trongate_administrators/login.

However, this approach had a known vulnerability: an unauthenticated request to trongate_administrators/logout would redirect to the secret login URL, revealing it in the HTTP response headers. The secret was exposed to anyone who knew where to look.

Why We Changed

The new login module takes a fundamentally different approach. Instead of hiding the login URL, it focuses on proper security controls:

  • Timing-safe password verification — Prevents enumeration attacks by always using a valid bcrypt hash, even when the username does not exist
  • Brute-force protection — Configurable rate limiting tracks failed attempts by both identifier and IP address, blocking access after a configurable number of failures
  • Dedicated login_attempts table — Failed attempts are tracked independently of user records, so even non-existent usernames are rate-limited
  • Configurable user levels — Support for multiple user tables (administrators, members, etc.) from a single module
  • Multi-identifier login — Users can log in with any configured identifier (username, email, or both)

How It Works Now

The login module is accessed at the public URL login/login. A discoverable, well-audited login page with proper security controls is far more robust than a hidden one that can be leaked through redirects.

Migrating to the New System

If you are upgrading an existing project, please refer to the Migrating to the Universal Login Module guide for step-by-step instructions.

Benefits of the New Approach

  • No secret URLs to leak — Security is built into the authentication process, not the URL
  • Proper rate limiting — Blocks brute-force attempts regardless of whether the username exists
  • Consistent authentication — All user types authenticate through the same robust system
  • Forgot password flow — Built-in password reset with SMTP-based email delivery
  • Configurable — Adjust security settings without touching code
