[meta-description]Trongate v2 gives you full control over how your form validation errors are rendered. In this page you'll learn how to change the appearance of validation errors by adding or adjusting some simple CSS rules.[/meta-description]

Styling Validation Errors

The Trongate framework includes a core CSS file named trongate.css. While there is no obligation to include this stylesheet on every page, leveraging it allows you to quickly build responsive, professional-looking webpages that handle form states out of the box.

How Form Validation Errors Are Rendered

When working with Trongate, there are two primary ways that form validation errors are rendered:

  1. Summary validation errors (standard)
  2. Inline validation errors

In both instances, validation errors are rendered as unordered lists (using <ul> tags), with individual error messages appearing as list items (using <li> tags).

1. Summary Validation Errors

Summary errors are typically positioned above the form or near the top of the page. In this mode, all validation messages are grouped into a single list, providing the user with a comprehensive overview of what needs to be corrected.

Below is an example of the HTML source code generated for a summary error display:

[code] <ul class="validation-errors validation-errors--summary"> <li>The task title field is required.</li> <li>The description field is required.</li> </ul> <form action="http://localhost/app/tasks/submit" method="post" class="highlight-errors shake"> <label>Task Title</label> <input placeholder="Task title..." autocomplete="off" type="text" name="task_title" value=""> <label>Description</label> <textarea placeholder="Enter task details here..." name="description"></textarea> <label> <input type="checkbox" name="complete" value="1">Task complete </label> <div class="text-center"> <a href="http://localhost/app/tasks/manage" class="button alt">Cancel</a> <button type="submit" name="submit" value="Submit">Submit</button> </div> <input type="hidden" name="csrf_token" value="2ad4deae80eb8d33793436478b93a3f2e9ac086da100ce74ef429f670dcf25e8"> </form> [/code]

2. Inline Validation Errors

With inline validation, error messages are displayed directly adjacent to the specific form field that failed validation. This provides immediate context to the user. Additionally, Trongate automatically applies the form-field-validation-error class to the invalid input or textarea, and adds a shake class to the opening <form> tag to grab the user's attention.

Here is the typical source code rendered for inline validation:

[code] <form action="http://localhost/app/tasks/submit" method="post" class="highlight-errors shake"> <label>Task Title</label> <ul class="validation-errors validation-errors--inline"> <li>The task title field is required.</li> </ul> <input placeholder="Task title..." autocomplete="off" type="text" name="task_title" value="" class="form-field-validation-error"> <label>Description</label> <ul class="validation-errors validation-errors--inline"> <li>The description field is required.</li> </ul> <textarea placeholder="Enter task details here..." name="description" class="form-field-validation-error"></textarea> <label> <input type="checkbox" name="complete" value="1">Task complete </label> <div class="text-center"> <a href="http://localhost/app/tasks/manage" class="button alt">Cancel</a> <button type="submit" name="submit" value="Submit">Submit</button> </div> <input type="hidden" name="csrf_token" value="2ad4deae80eb8d33793436478b93a3f2e9ac086da100ce74ef429f670dcf25e8"> </form> [/code]

In order for inline validation errors to be rendered, a class of highlight-errors should be added to your opening form tag.

In addition, the validation_errors() function should be applied adjacent to the relevant form fields. For example:

[code=vf] <?php echo form_open($form_location, array('class' => 'highlight-errors')); echo form_label('Email Address'); echo validation_errors('email'); echo form_input('email', $email); echo form_label('Password'); echo validation_errors('password'); echo form_password('password'); echo form_submit('submit', 'Register'); echo form_close(); ?> [/code]

For more information about this, see: Displaying Validation Errors.

Customizing the Appearance

Both summary and inline error lists share a base class of validation-errors, which handles the general layout, borders, and colors. Specific modifiers are then added depending on the error type:

  • .validation-errors: The base class applied to all error <ul> elements.
  • .validation-errors--summary: Added when errors are grouped at the top of the form.
  • .validation-errors--inline: Added when errors are placed next to specific fields.
  • .form-field-validation-error: Added directly to <input> or <textarea> elements that have failed validation.

The Default CSS

By default, trongate.css uses CSS variables and animations to highlight issues. Here are the core rules that govern the look and feel of validation errors:

[code=css] form .form-field-validation-error, form .form-field-validation-error:focus { border: 3px solid var(--danger); background-color: #ff000011; } .validation-errors { margin: 0.5em 0 1em 0; padding: 0.3em 1em; border: 1px solid var(--danger); background-color: #ff000011; color: var(--danger); border-radius: 6px; font-size: .9em; } .validation-errors li { margin-left: 1.2em; padding-left: 0; } .shake { animation: form-shake 0.35s ease-in-out; } @keyframes form-shake { 0% { transform: translateX(0); } 20% { transform: translateX(-6px); } 40% { transform: translateX(6px); } 60% { transform: translateX(-4px); } 80% { transform: translateX(4px); } 100% { transform: translateX(0); } } [/code]

Practical Examples: Customizing List Markers

One of the easiest ways to change the "vibe" of your validation errors is by tweaking the list markers. By targeting the .validation-errors--inline class, you can quickly move away from the default look. Here are a few ways to style your error lists using simple CSS:

Minimalist (No Markers)

A clean, modern look that relies purely on text and color contrast.

[code=css] .validation-errors--inline { list-style-type: none; } [/code]

Professional Square

A square marker often feels more like a "UI component" than a document list. This is a great choice for admin panels.

[code=css] .validation-errors--inline { list-style-type: square; } [/code]

Modern Dash Style

Using the ::marker pseudo-element, you can replace standard bullets with a clean dash.

[code=css] .validation-errors--inline { list-style-type: none; padding-left: 0; } .validation-errors--inline li::marker { content: "– "; } [/code]

Numbered Errors (Decimal)

Useful if multiple validation rules per field are common, helping the user address them sequentially.

[code=css] .validation-errors--inline { list-style-type: decimal; list-style-position: inside; } [/code]

Using Icons and Symbols

Sometimes a standard bullet point doesn't convey enough urgency. You can use CSS to replace list markers with symbols or emojis to give your validation errors more visual impact.

1. The "Warning" Emoji

Using a standard ::marker is the most modern way to inject an emoji into your error list.

[code=css] .validation-errors--inline li::marker { content: "⚠️ "; } [/code]

2. The "Error" Cross

A simple "X" symbol is a universal indicator that something needs to be fixed. It works perfectly with the default red theme.

[code=css] .validation-errors--inline li::marker { content: "❌ "; font-size: 0.8em; } [/code]

3. The "Action" Pointer

Guide the user's eye directly to the input field with a pointer emoji.

[code=css] .validation-errors--inline li::marker { content: "👉 "; } [/code]

4. Custom "Check" (For success states)

If you've created a custom .validation-success class for valid fields, you can use a green checkmark.

[code=css] .validation-success li::marker { content: "✅ "; } [/code]

Advanced: Using Pseudo-elements for Precision

If you need more control over positioning, use the ::before pseudo-element. This allows the icon to be styled independently of the text.

[code=css] .validation-errors--inline { list-style-type: none; padding-left: 0; } .validation-errors--inline li::before { content: "●"; color: var(--danger); font-weight: bold; display: inline-block; width: 1em; margin-left: -1em; } [/code]

Improving Readability with Spacing

When using large icons, emojis, or multi-line error messages, the list items can sometimes appear crowded. You can use the margin-top property on the li elements to create "breathing room" between each error.

Adding Vertical Spacing

This approach is particularly helpful when using emojis like ⚠️ or ❌, as it prevents the icons from overlapping or feeling too tight.

[code=css] /* Add a small gap between individual error messages */ .validation-errors li { margin-top: 0.8em; } /* Ensure the first error doesn't have extra top spacing */ .validation-errors li:first-child { margin-top: 0; } [/code]

Adjusting Line Height

If your error messages are long and wrap onto multiple lines, increasing the line-height can also make the text much easier to scan.

[code=css] .validation-errors li { line-height: 1.5; margin-bottom: 0.5em; /* An alternative to margin-top */ } [/code]

Note: In the example above, we targeted .validation-errors li. This will apply the spacing to both summary and inline validation lists, ensuring a consistent look across your entire application.

When using emojis as markers, ensure your meta charset is set to UTF-8 in your HTML header to ensure they render correctly across all devices.
