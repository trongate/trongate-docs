The Complete Token Lifecycle

Trongate's token system follows a simple, predictable flow. No hidden state. No magic middleware. Just explicit steps that you control. This page explains exactly how the token system operates for authentication and authorization.

If the information on this page seems overwhelming, don't worry. Remember, the framework does all of the heavy lifting. Making the whole thing work is remarkably easy.

Here's what happens from login to logout:

  1. User logs in - They submit credentials (username/password, email/password, etc.)
  2. You validate credentials - Check against YOUR database table (e.g., members, administrators)
  3. Token gets generated - A random 32-character string is created
  4. Token gets stored in database - Insert into trongate_tokens with user_id and expiry_date
  5. Token gets stored on device - Save in session or cookie
  6. User makes subsequent requests - Token is sent automatically with each request
  7. Token gets validated - Check database: does it exist? Has it expired? Does user have required level?
  8. Access granted or denied - If valid, proceed. If not, redirect to login
  9. Token expires or gets destroyed - Either reaches expiry_date or user logs out

What Trongate Does (and Doesn't Do)

Trongate's token system is managed via a module, that's included with the framework, named 'Trongate Tokens'.

The Trongate Tokens module is responsible for:

  • Generating tokens
  • Storing tokens
  • Validating tokens
  • Destroying tokens

That's all.

The Trongate Tokens module is not responsible for:

  • How users log in
  • How you manage the login process
  • Any kind of rate limiting
  • Restricting how many users are allowed to be logged in

Once you've decided a user is legitimate, THEN you call Trongate to generate a token.

This separation is intentional. It keeps the token system simple, flexible, and future-proof.

Trongate doesn't care HOW users log in. Username/password? Email verification? OAuth? Two-factor? Biometrics? Face recognition? Magic links? Up to you!

Example: Members-Only Website

Let's walk through a concrete example. You're building a members-only website.

Setup Phase

First, create your database tables:

[code=sql] -- Your application table CREATE TABLE members ( id INT NOT NULL PRIMARY KEY AUTO_INCREMENT, email VARCHAR(125) UNIQUE NOT NULL, password VARCHAR(60) NOT NULL, first_name VARCHAR(50), trongate_user_id INT(11), FOREIGN KEY (trongate_user_id) REFERENCES trongate_users(id) ); -- Trongate's security tables (already covered in previous page) -- trongate_user_levels -- trongate_users -- trongate_tokens [/code]

Add a "member" user level:

[code=sql] INSERT INTO trongate_user_levels (id, level_title) VALUES (2, 'member'); [/code]

Registration Phase

When someone registers:

[code=php] // 1. Create trongate_users record $user_data = [ 'code' => make_rand_str(32), 'user_level_id' => 2 // member level ]; $trongate_user_id = $this->db->insert($user_data, 'trongate_users'); // 2. Create members record $member_data = [ 'email' => $posted_email, 'password' => password_hash($posted_password, PASSWORD_DEFAULT), 'first_name' => $posted_first_name, 'trongate_user_id' => $trongate_user_id ]; $this->db->insert($member_data, 'members'); [/code]

Login Phase

When someone logs in:

[code=php] // 1. Validate credentials (YOUR code) $member = $this->db->get_where_custom('email', $posted_email, 'members'); if (!$member || !password_verify($posted_password, $member->password)) { // Invalid credentials - show error redirect('login'); } // 2. Generate token $token_data = [ 'user_id' => $member->trongate_user_id, 'expiry_date' => time() + (86400 * 30) // 30 days ]; $token = $this->trongate_tokens->generate_token($token_data); // 3. Redirect to members area redirect('members/dashboard'); [/code]

Protected Page Phase

On any members-only page:

[code=php] public function dashboard(): void { // Validate token $token = $this->trongate_tokens->attempt_get_valid_token(2); // level 2 = member if ($token === false) { redirect('login'); } // Token valid - show dashboard $data['view_file'] = 'dashboard'; $this->templates->members_area($data); } [/code]

The example above assumes you have a members_area template created.

Logout Phase

[code=php] public function logout(): void { // Destroy token (clears session, cookies, database, and expired tokens) $this->trongate_tokens->destroy(); redirect('welcome'); } [/code]

The destroy() method automatically cleans up expired tokens from the database as part of the logout process. You don't need to call delete_old_tokens() separately unless you're running a scheduled maintenance task.

Token Storage Locations

When you generate a token, it gets stored in TWO places:

  1. Database - Always stored in trongate_tokens table
  2. User's device - Stored as either a session variable OR a cookie

Storage Option 1: Session (Default)

[code=php] $token_data = ['user_id' => $trongate_user_id]; $token = $this->trongate_tokens->generate_token($token_data); // Token stored in $_SESSION['trongatetoken'] [/code]

Use case: Standard web applications. Token persists until browser closes or session expires.

Storage Option 2: Cookie

[code=php] $token_data = [ 'user_id' => $trongate_user_id, 'expiry_date' => time() + (86400 * 30), // 30 days 'set_cookie' => true ]; $token = $this->trongate_tokens->generate_token($token_data); // Token stored in $_COOKIE['trongatetoken'] [/code]

Use case: "Remember me" functionality. Token persists across browser sessions.

Token Validation Priority

When validating tokens, Trongate checks two locations in this order:

  1. Cookies - $_COOKIE['trongatetoken']
  2. Session - $_SESSION['trongatetoken']

The first valid token found wins.

Token Expiration

Every token has an expiry_date stored as a Unix timestamp.

Default lifespan: 86400 seconds (24 hours)

When validating tokens, Trongate automatically checks:

[code=sql] SELECT * FROM trongate_tokens WHERE token = ? AND expiry_date > ? -- Must be greater than current time [/code]

Expired tokens return false during validation, forcing re-authentication.

The Trongate Security Module

Trongate includes a Trongate_security module that provides a centralized way to handle authorization checks across your application.

Instead of calling trongate_tokens methods directly, you can use:

[code=php] public function dashboard(): void { // Centralized security check $this->trongate_security->make_sure_allowed('members area'); // If we reach here, user is authorized $data['view_file'] = 'dashboard'; $this->templates->public($data); } [/code]

The Trongate_security module uses scenario-based access control, making it easy to define different authorization rules for different parts of your application (admin panel, members area, etc.).

Using Trongate_security is optional but recommended. It provides a clean separation between your application logic and security checks. We'll cover this in detail in a later section.

Security Considerations

A few important points:

  • Tokens are NOT encrypted - They're random strings. Treat them like passwords.
  • Always use HTTPS - Tokens sent over HTTP can be intercepted.
  • Hash passwords properly - Use password_hash() and password_verify(). Never store plain text.
  • Set appropriate expiry dates - Short-lived tokens for sensitive operations, longer for convenience.
  • Clean up on user deletion - When deleting a user, delete their tokens too using delete_old_tokens().

Database Requirements

Trongate's token system requires a MySQL or MariaDB database.

The system is database-driven by design. Every token validation queries the trongate_tokens table to check:

  • Does the token exist?
  • Has it expired?
  • What user level does it have?

This approach ensures tokens can be invalidated instantly (e.g., when a user logs out or gets banned) without waiting for session timeouts.
