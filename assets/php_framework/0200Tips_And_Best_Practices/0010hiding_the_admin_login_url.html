Hiding The Admin Login URL — Deprecated

Note: This page describes a technique used by previous versions of Trongate. The Universal Login Module (shipped with Trongate v2) replaces this approach entirely. Security through obscurity is no longer recommended — the new login module is secure by design.

The Old Approach

In previous versions of Trongate, the trongate_administrators module featured a configurable $secret_login_segment property. Administrators could set a private word that acted as the login URL, making it harder for attackers to find the login page. Custom routing mapped this secret word to trongate_administrators/login.

However, this approach had a known vulnerability: an unauthenticated request to trongate_administrators/logout would redirect to the secret login URL, revealing it in the HTTP response headers. This meant the secret was exposed to anyone who knew where to look.

Why We Changed

The new Universal Login Module takes a different approach. Instead of hiding the login URL, it focuses on:

  • Secure credential validation — Timing-safe hash comparison prevents enumeration attacks
  • Brute-force protection — Configurable rate limiting blocks repeated failed attempts
  • Configurable view styles — Three different login page layouts to suit your site's design
  • Multi-table support — Authenticate administrators, members, or any user type from one module

How It Works Now

The login module is accessed at the public URL login/login. This is intentional — a discoverable, well-audited login page with proper security controls is more robust than a hidden one that can be leaked through a redirect.
